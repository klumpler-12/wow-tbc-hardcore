Roadmap

From concept to CurseForge. Track every phase of the Hardcore Plus addon as we build it.

✓ Done In Progress ○ Pending
  • ✓
    Feature specification
    • Core feature list finalized
    • Mode definitions (Hardcore / Hybrid / Null)
    • Punishment & reward catalog drafted
  • ✓
    Technical architecture
    • Module dependency graph
    • Data flow diagrams
    • Addon communication protocol spec
  • Requirements mapping
    • OnlyFangs 3 specific requirements
    • API surface definition
    • SavedVariables schema design
Dev Note Architecture is modular by design so guilds can enable/disable individual systems. Priority is getting the death detection and rule engine stable before anything else.
  • ○
    Lua addon skeleton
    • TOC file and folder structure
    • Core module loader
    • Event registration framework
  • ○
    SavedVariables structure
    • Per-character data schema
    • Guild-wide shared state
    • Migration and versioning system
  • ○
    Addon communication protocol
    • Prefix registration
    • Message serialization / deserialization
    • Throttling and queue management
  • ○
    Basic death detection
    • PLAYER_DEAD / PLAYER_ALIVE_AFTER_RELEASE hooks
    • Ghost state tracking
    • Death context capture (zone, level, killer)
Dev Note Death detection is the single most critical piece. Must handle edge cases: disconnect deaths, instance resets, battleground deaths (excluded), and spirit healer interactions.
  • ○
    Configurable ruleset system
    • Rule definition DSL
    • Condition evaluator (level ranges, zones, group size)
    • Action dispatcher (punish, reward, announce)
  • ○
    GM control panel
    • In-game rule editor UI
    • Permission levels (GM, Officer, Member)
    • Live rule preview and testing
  • ○
    Rule import/export
    • Serialized ruleset strings
    • Preset library (Strict HC, Hybrid, Casual)
    • Community sharing format
Dev Note The rule engine is the core differentiator. It needs to be flexible enough for any guild's ruleset while being simple enough for non-technical GMs to configure.
  • ○
    Death tracking module
    • Death log with full context
    • Death location recording (zone, coords)
    • Kill attribution (mob, player, environment)
  • ○
    Achievement system
    • Micro-achievement definitions (1000+ possible)
    • Progress tracking and toast notifications
    • Achievement point aggregation
  • ○
    Scoring engine (both modes)
    • Survival-time scoring (HC mode)
    • Weighted-points scoring (Hybrid mode)
    • Score normalization and decay
  • ○
    Leaderboards
    • Per-guild leaderboard frame
    • Sortable columns (score, deaths, achievements)
    • Historical snapshots and trends
Dev Note Scoring needs careful balancing. Survival-time scoring is straightforward but weighted-points needs tuning. Plan to ship with conservative defaults and expose multipliers to GMs.
  • ○
    House creation / management
    • House naming, colors, and emblems
    • Member assignment and roster management
    • House chat channel integration
  • ○
    Team aggregation
    • House score = sum of member scores
    • Weighted contribution tracking
    • Inactive member handling
  • ○
    House leaderboards
    • Head-to-head house comparison
    • Seasonal standings and history
    • House rivalry announcements
  • ○
    Draft system (optional)
    • Snake draft or random assignment
    • Captain pick mode
    • Trade window for member swaps
Dev Note Houses are the social glue. Draft system is stretch goal but even basic random assignment creates immediate team identity and rivalry.
  • ○
    Creative punishment triggers
    • Death-based triggers (first death, streak break)
    • Context-aware severity (dungeon wipe vs. world death)
    • Punishment queue and cooldown system
  • ○
    Reward distribution
    • Milestone reward checks
    • Streak bonus multipliers
    • House-wide reward events
  • ○
    RP-walk, gear lock mechanics
    • Forced RP-walk speed enforcement
    • Gear slot locking (unequip penalties)
    • Shame announcements in guild chat
Dev Note Punishments must be fun, not frustrating. The RP-walk and gear lock are crowd favorites from testing. Shame announcements add the social pressure that makes HC exciting.
  • ○
    Electron companion app
    • Real-time SavedVariables sync
    • Desktop notifications for deaths and events
    • Overlay / OBS integration for streamers
  • ○
    Guild site templates
    • Static site generator for guild pages
    • Auto-generated leaderboards and death feeds
    • Custom theme support
  • ○
    Character profiles
    • Armory-style character pages
    • Achievement showcase and death history
    • Shareable profile links
  • ○
    Analytics dashboard
    • Death heatmaps by zone
    • Class/spec survival statistics
    • Guild activity trends and engagement metrics
Dev Note The companion app is ambitious but the streamer overlay alone could drive adoption. Guild sites should be zero-config: install addon, data appears on your page.
  • ○
    Beta testing
    • Closed alpha with core testers
    • Open beta with partner guilds
    • Stress testing addon communication at scale
  • ○
    CurseForge packaging
    • Automated build pipeline
    • Version tagging and changelog generation
    • CurseForge listing and metadata
  • ○
    Documentation
    • GM setup guide
    • Player quick-start guide
    • API reference for advanced users
  • ○
    Community onboarding
    • Discord server setup and bot integration
    • Tutorial videos and walkthroughs
    • Launch event with partner guilds
Dev Note Launch timing ideally coincides with a new OnlyFangs season. CurseForge listing is critical for discoverability. Discord bot can mirror in-game death feeds.
